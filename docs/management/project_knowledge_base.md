## 2025年8月17日: NextAuth認証システム復旧作業の知見

### 作業概要
Next.jsアプリケーションで発生していた認証関連のエラー（ハイドレーションエラー、無限ループ、500エラー）を解決し、NextAuth.jsの認証システムを正常に動作させる作業を実施しました。

### 発生していた問題と解決策

1.  **ハイドレーションエラー**
    *   **症状**: `Hydration failed because the initial UI does not match what was rendered on the server`
    *   **原因**: `useSession` フックの初期状態とサーバーサイドレンダリングの不一致
    *   **対応策**: `src/app/page.tsx` および `src/components/AuthenticatedLayout.tsx` に `mounted` 状態を追加し、クライアントサイドでのみレンダリングされるように修正。

2.  **認証の無限ループ**
    *   **症状**: ログイン後に `http://localhost:3000/auth/login?callbackUrl=...` に自己回帰
    *   **原因**: NextAuth.jsの `NEXTAUTH_URL` 環境変数の設定ミスとポート不一致、および手動リダイレクトの競合
    *   **対応策**: `NEXTAUTH_URL=http://localhost:3000` を設定。`src/components/AuthForm.tsx` の `signIn` 関数で `redirect: true` に変更し手動リダイレクトを削除。`src/components/AuthenticatedLayout.tsx` の手動リダイレクトを削除。`src/lib/auth.ts` に `callbacks.redirect` を追加し、認証成功後のリダイレクトを明示的に制御。

3.  **500 Internal Server Error**
    *   **症状**: 静的ファイル（JavaScriptチャンク）が見つからない
    *   **原因**: `next.config.js` の `output: 'standalone'` 設定が開発環境で問題を引き起こす
    *   **対応策**: `next.config.js` の `output: 'standalone'` を開発環境用にコメントアウト。

4.  **TypeScript型エラー**
    *   **症状**: 60件以上のTypeScriptエラーが発生
    *   **原因**: NextAuth.jsの型定義が不適切、セッション型の拡張が不足
    *   **対応策**: `src/lib/auth.ts` に NextAuth.jsの型拡張を追加。`@types/bcrypt` をインストール。

### 認証フローの最適化

-   **開発環境でのダミー認証**: `src/lib/auth.ts` にて、開発環境 (`process.env.NODE_ENV === 'development'`) の場合に `CredentialsProvider` が常に認証成功を返すように修正。これにより、開発効率が向上。

### 技術的な改善点

1.  **セッション状態の安全な処理**: `mounted` 状態によるクライアントサイドでのみのレンダリング、セッション状態の段階的なチェック、エラー状態の適切な処理。
2.  **認証フローの最適化**: NextAuth.jsの標準的な認証フローへの統一、手動リダイレクトの削除、コールバックURLの適切な設定。
3.  **型安全性の向上**: NextAuth.jsの型定義の拡張、セッションとユーザー情報の型安全性確保、TypeScriptエラーの完全解決。

### 現在の動作確認状況

-   アプリケーション: `http://localhost:3000` で正常稼働中。
-   認証システム: NextAuth.jsの正常動作、開発環境でのダミーログイン機能、セッション管理の正常動作、ログアウト機能。
-   ユーザーインターフェース: 認証状態の表示、ログイン/ログアウトフォーム、認証済みユーザー向けダッシュボード、ナビゲーションバー。
-   開発環境: ポート3000での正常起動、ホットリロードの正常動作、TypeScriptエラーの解決、ビルドプロセスの正常化。

### 発生した問題と対応策 (2025年8月17日)

1.  **Tailwind CSS PostCSS プラグインエラー**
    *   **問題**: `npm run dev` 実行時に「It looks like you're trying to use `tailwindcss` directly as a PostCSS plugin.」というエラーが発生し、ビルドが停止した。これは、Tailwind CSS のバージョンアップに伴い、PostCSS 設定で `tailwindcss` プラグインの指定方法が変更されたため。
    *   **対応策**: 
        1.  `@tailwindcss/postcss` パッケージをインストールする。
            `npm install -D @tailwindcss/postcss`
        2.  `postcss.config.js` の `plugins` 設定を `tailwindcss: {}` から `'@tailwindcss/postcss': {}` に変更する。
    *   **関連ファイル**: `postcss.config.js`, `package.json`
    *   **知見**: Tailwind CSS のメジャーバージョンアップ時には、PostCSS プラグインの指定方法が変更される可能性があるため、公式ドキュメントを確認し、必要なパッケージのインストールと設定ファイルの更新を行う必要がある。

2.  **Google Fonts ダウンロード失敗 (AbortError)**
    *   **問題**: `npm run dev` 実行時に「Failed to download `Noto Sans JP` from Google Fonts. Using fallback font instead.」というエラーが繰り返し発生した。これは、Next.js の `@next/font/google` が Google Fonts からフォントファイルをダウンロードする際に、ネットワークの問題（タイムアウトなど）によりリクエストが中断されたため。
    *   **対応策**: 
        *   一時的なネットワークの問題である可能性が高いため、開発サーバーを再起動するか、ネットワーク環境を確認する。
        *   恒久的な解決策としては、フォントファイルをローカルにホストするか、CDN の安定性を確認する。
    *   **関連ファイル**: `src/app/layout.tsx` (フォント設定箇所)
    *   **知論**: 開発環境での Google Fonts のダウンロード失敗は、ネットワーク環境に依存するため、本番環境での動作とは異なる場合がある。表示崩れの原因となるため、開発時には注意が必要。

3.  **Google Fonts の一時的な無効化**
    *   **問題**: `npm run dev` 実行時に Google Fonts (`Noto Sans JP`, `Shippori Mincho B1`) のダウンロードが `AbortError` で失敗し、ビルドは完了するもののフォントが適用されない問題が発生した。これはネットワーク接続の問題に起因する。
    *   **対応策**: `src/app/layout.tsx` 内の Google Fonts のインポートと設定を一時的にコメントアウトし、フォントのダウンロードを無効化した。これにより、ビルドエラーは解消され、アプリケーションの表示が可能になった。
    *   **関連ファイル**: `src/app/layout.tsx`
    *   **知見**: 開発環境での Google Fonts のダウンロード問題は、ネットワーク環境に大きく依存するため、開発の継続性を優先して一時的に無効化することも有効な手段である。本番環境では安定したCDN経由での提供を検討する必要がある。

4.  **`docs` ディレクトリの競合**
    *   **問題**: TypeScript の設定において、`docs` ディレクトリ内のファイルがビルドプロセスに影響を与え、エラーを引き起こす可能性があった。
    *   **対応策**: `tsconfig.json` の `exclude` 設定に `docs/**/*` を追加し、`docs` ディレクトリ内のファイルを TypeScript のコンパイル対象から除外した。
    *   **関連ファイル**: `tsconfig.json`
    *   **知見**: ドキュメントや設定ファイルなど、ソースコードではないディレクトリは、ビルドプロセスに不要な影響を与えないよう、TypeScript の `exclude` 設定や `.gitignore` で適切に除外することが重要である。

5.  **Missing dependencies (不足している依存関係)**
    *   **問題**: `react-markdown`, `rehype-raw`, `rehype-sanitize`, `remark-gfm` といったコンポーネントやライブラリが不足しており、ビルドエラーの原因となっていた。
    *   **対応策**: 以下のコマンドで不足している依存関係をインストールした。
        `npm install react-markdown rehype-raw rehype-sanitize remark-gfm`
    *   **関連ファイル**: `package.json`
    *   **知見**: プロジェクトの移行やクローン時には、`package.json` に記載されている依存関係がすべてインストールされているか、`npm install` または `yarn install` を実行して確認することが不可欠である。特に、特定の機能が動作しない場合は、関連する依存関係の不足を疑うべきである。

## 2025年8月17日: 新規記事投稿APIの不具合対応 (TSK-036) からの知見

### 1. インシデント概要

- **問題:** 新規記事投稿ページで、ヒーロー画像が保存されず、下書き機能が動作しない不具合が発生。
- **根本原因:** フロントエンドとバックエンド間でのAPI契約の不一致。具体的には、リクエストペイロードのJSONキー名が異なっていた (`heroImage` vs `heroImageUrl`)。

### 2. 教訓と再発防止策

#### a. API契約の文書化と合意形成の徹底

- **教訓:** 口頭や非公式な合意のみで開発を進めると、フロントエンドとバックエンドの実装に乖離が生まれるリスクが非常に高い。
- **対策:** APIを新規設計・変更する際は、必ず**API仕様書**（エンドポイント、HTTPメソッド、リクエスト/レスポンスのペイロード、ステータスコードなど）をドキュメントとして作成する。そして、関係者（フロントエンド担当、バックエンド担当）がその仕様書に合意してから実装に着手するプロセスを徹底する。

#### b. 型定義の共有による契約のコードレベルでの強制

- **教訓:** 人間の注意力には限界があり、タイポや記憶違いによるキー名の不一致は容易に発生する。
- **対策:** 可能であれば、APIのデータ構造に関する型定義（TypeScriptの`interface`や`type`）を、フロントエンドとバックエンドで共有する。
    - 例えば、`src/types/api/article.ts` のようなファイルを作成し、`ArticleCreationRequest` のような型を定義する。
    - これにより、コンパイル時に型の不一致を検知でき、実行時エラーを未然に防ぐことができる。

#### c. 開発サイクルの最終段階における統合テストの必須化

- **教訓:** 各モジュール（フロントエンド、バックエンド）の単体テストだけでは、モジュール間の連携ミスは見逃される。
- **対策:** 機能開発の完了を判断する前に、必ずフロントエンドとバックエンドを結合した状態での**統合テスト**（E2Eテスト、または手動でのシナリオテスト）を実施する。これにより、「フォームに入力してボタンを押し、データが正しくDBに保存される」という一連の流れ全体が正しく機能することを確認する。

## 2025年8月17日: daisyUIテーマ適用問題の解決 (TSK-039) からの知見

### 1. インシデント概要

- **問題:** `daisyUI` のカスタムテーマを適用しようとしたが、関連ファイルの複雑な依存関係により、ローカル環境でレイアウトが正しく表示されない問題が発生した。
- **解決策:** ユーザーとCursorの協力により、設定の競合を解消し、UIの完全な再現に成功した。

### 2. 問題の核心と解決策

#### a. 原因: TailwindCSSのバージョン競合
- **詳細:** プロジェクト内で、TailwindCSS v4系で推奨されるPostCSS設定 (`@tailwindcss/postcss`) と、旧来のv3系設定 (`tailwindcss: {}`) が混在していた。これがスタイルの計算を阻害する最大の原因だった。

#### b. 解決策: バージョンと設定の完全統一
1.  **PostCSS設定の統一:** `postcss.config.js` を、`daisyUI` が安定して動作する旧来のv3系互換の設定に統一した。
2.  **TailwindCSSのダウングレード:** 関連するTailwindCSSのパッケージをv3系にダウングレードし、バージョン間の不整合を解消した。
3.  **依存関係のクリーンアップ:** `npm uninstall` を使用して競合する可能性のあるパッケージ (`@tailwindcss/postcss` など) を完全に削除し、`node_modules` と `package-lock.json` を再生成して、クリーンな依存関係を確保した。

### 3. 教訓

- **CSSフレームワークのバージョン管理:** `TailwindCSS` のようなビルドプロセスに深く関わるライブラリでは、メジャーバージョン間の差異が破壊的な影響を及ぼすことがある。プロジェクト全体でバージョンを統一し、関連する全てのツール（PostCSSプラグインなど）もそのバージョンに対応していることを確認することが極めて重要である。
- **設定のブラックボックス化を防ぐ:** スタイルが適用されない場合、コンポーネントのコードだけでなく、`tailwind.config.js`, `postcss.config.js`, `package.json` といった設定ファイルを横断的に調査する必要がある。

## 2025年8月18日: 記事CRUD機能実装 (TSK-040) からの知見

### 1. 概要
記事管理(CRUD)機能の実装過程で、認証、データベース接続、APIレスポンス形式、フロントエンドのライブラリに関する複数の問題が顕在化し、それらを解決した。

### 2. 実装中の主要な問題と解決策

#### a. 認証コンテキストの不統一
- **問題:** ページやコンポーネントによって、NextAuth由来の `useSession` と、カスタムコンテキストの `useAuth` が混在していた。これにより、開発環境の認証スキップが一部で機能せず、「ログインが必要です」エラーが発生した。
- **解決策:** プロジェクト内の認証状態の取得を、すべてカスタムの `useAuth()` フックに統一した。
- **教訓:** 認証のような横断的な関心事は、プロジェクト全体で単一の信頼できる情報源（Single Source of Truth）から取得するべきである。複数の認証管理方法の混在は、混乱とバグの温床となる。

#### b. 開発環境のデータベース接続
- **問題:** `psql` でのユーザー作成失敗や、`prisma` コマンド実行時の `DATABASE_URL` 環境変数未定義エラーが発生した。
- **解決策:**
    1.  `brew services` でPostgreSQLの起動状態を確認。
    2.  `psql -d postgres` でデフォルトユーザーとして接続し、`CREATE USER` と `CREATE DATABASE` を実行して、アプリケーション用のユーザーとDBを準備した。
    3.  `npx prisma db push` が `.env.local` を読み込まない問題に対し、`DATABASE_URL="postgresql://..." npx prisma db push` のように、コマンドの接頭辞として環境変数を直接指定することで解決した。
- **教訓:** データベースのセットアップは、手順を明確に文書化することが重要。特に、ローカル環境で `prisma` のようなツールが環境変数を正しく読み込めないケースは頻出するため、直接指定する対処法は有効なデバッグ手段となる。

#### c. APIレスポンス形式の不一致
- **問題:** 記事作成API (`/api/articles`) が記事オブジェクトを直接返していたが、フロントエンドは `{ article: ... }` というラップされたオブジェクトを期待していたため、`slug` が読み取れずエラーとなっていた。
- **解決策:** APIのレスポンスを `return NextResponse.json({ article }, { status: 201 })` のように、フロントエンドの期待する形式に合わせて修正した。
- **教訓:** APIを設計・実装する際は、リクエストだけでなくレスポンスのデータ構造も明確に定義し、フロントエンドとバックエンドでその契約を遵守することが不可欠である。

#### d. `react-markdown` ライブラリの破壊的変更
- **問題:** `ReactMarkdown` コンポーネントに `className` プロパティを渡したところ、`Assertion: Unexpected 
`className` prop` というランタイムエラーが発生した。
- **解決策:** ライブラリのChangelogを確認し、`className` プロパティが廃止されたことを突き止めた。コンポーネントのルート要素へのクラス指定を削除し、`components` プロパティ経由で個別のHTML要素にスタイルを適用する方法に切り替えた。
- **教訓:** 外部ライブラリを導入・更新する際は、メジャーバージョンアップに伴う破壊的変更に特に注意が必要。エラーメッセージを精読し、公式ドキュメントやChangelogを確認する習慣が、迅速な問題解決に繋がる。
