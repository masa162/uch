# タスク詳細: TSK-001

**ID**: `TSK-001`
**タイトル**: 旧プロジェクト(uchi)からの移行計画策定
**ステータス**: 未着手
**優先度**: 高
**担当者**: Gemini CLI PM

## 1. タスクの目的

旧プロジェクト(`uchi`)の資産を分析し、「再利用するもの」と「新規に構築するもの」を明確に切り分ける。その上で、クリーンで安定した新プロジェクト(`uch`)の開発計画を策定し、関係者で合意する。

## 2. 背景

開発が停滞した旧プロジェクト`uchi`の資産を、エラーや負の遺産を持ち込まずに、新プロジェクト`uch`で最大限活用するため。

## 3. 具体的な作業内容 (PM担当)

### 3.1. 旧リポジトリ(`uchi`)の分析

*   **ソースコード分析**:
    *   実装済みの機能範囲（認証、投稿など）とコードの品質を評価する。
    *   再利用可能なコンポーネントやロジックを特定する。
*   **データベース設計分析**:
    *   `prisma/schema.prisma`を基に、データモデルを評価する。
*   **インフラ設定分析**:
    *   `docker-compose.yml`, `Dockerfile`, VPS関連ドキュメントを評価する。

### 3.2. プロジェクトオーナーへのヒアリング

以下の点について、オーナーの意向を確認する。

*   **中核機能**: `uchi`で実装した機能のうち、「絶対に必要」なものは何か？
*   **不要/改修機能**: 不要、または作り直したい機能は何か？
*   **進捗の定義**: `https://uchinokiroku.com/` の進捗とは、具体的にどの機能がどのレベルまで到達していた状態か？

### 3.3. 成果物の作成

*   **移行計画書**:
    *   再利用する機能/コンポーネントのリスト。
    *   新規に開発する機能のリスト。
    *   新しいデータベースの設計案。
    *   VPS環境のクリーンアップと再構築の手順案。
*   **開発タスクリスト**:
    *   上記計画書に基づき、具体的な開発タスクを洗い出し、`task_log.csv`に登録する。

## 4. 完了の定義

*   「移行計画書」が完成し、プロジェクトオーナーの承認を得る。
*   初期の開発タスクが`task_log.csv`に登録される。
