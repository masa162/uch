# デプロイ関連 (Docker, VPS) 知見集

## 2025年8月31日: デプロイ成功のための必須ファイル (TSK-095)

### 概要
デプロイプロセスを安定させるため、`Dockerfile`を修正し、本番用Dockerイメージに必要なファイルがすべて含まれるように改善した。

### 問題
従来の`Dockerfile`では、Next.jsのビルド成果物のみが最終イメージにコピーされていた。そのため、本番環境で`docker exec`を使ってコンテナに入り、Prismaのマイグレーションコマンド (`npx prisma migrate deploy`) を実行しようとすると、`prisma/schema.prisma`やマイグレーションファイルが見つからずに失敗していた。

### 解決策
`Dockerfile`の最終ステージに、`prisma`ディレクトリをコピーする`COPY`命令を追加した。

```Dockerfile
# (前略)
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static

# Prismaスキーマとマイグレーションファイルをコピー（データベースマイグレーション用）
COPY --from=builder --chown=nextjs:nodejs /app/prisma ./prisma

# (後略)
```

### 教訓
- **Dockerイメージには実行時に必要なファイルをすべて含める**: Next.jsアプリケーションの実行自体には不要でも、データベースマイグレーションのような運用上必要なコマンドが依存するファイル（`prisma`ディレクトリなど）は、すべて最終イメージに含める必要がある。
- **手動コピーからの脱却**: この修正により、デプロイごとに`docker cp`コマンドで手動でファイルをコンテナにコピーする手間がなくなり、デプロイプロセスがよりシンプルで再現性の高いものになった。

## 2025年8月18日: 本番環境でのPrismaマイグレーション安定化 (TSK-046) からの知見

### 1. 概要
本番環境でのPrismaマイグレーションの安定化作業において、当初想定されたマイグレーション失敗ではなく、Dockerビルド時のGoogle Fontsタイムアウトが根本原因であることが判明した。この問題解決と、デプロイプロセスの完全自動化検証を通じて得られた知見を記録する。

### 2. 主要な問題と解決策

#### a. Dockerビルド時のGoogle Fontsタイムアウト
- **問題:** `docker compose up --build -d` 実行時に、Next.jsのビルドプロセスがGoogle Fontsのダウンロードでタイムアウトし、ビルドが失敗する。
- **原因:** `src/app/layout.tsx` で `Shippori_Mincho_B1` フォントをGoogle Fontsからインポートしており、ビルド環境からのアクセスが不安定だったため。
- **解決策:** `src/app/layout.tsx` から `Shippori_Mincho_B1` のインポートを削除し、代替のフォント設定（`Noto_Sans_JP` のみ、またはローカルフォント）に切り替えた。
- **教訓:** Dockerビルド環境では、外部リソース（特にWebフォントなど）へのアクセスが不安定になる場合がある。ビルドの安定性を確保するため、可能な限りローカルにホストするか、ビルド時に外部アクセスを必要としない設定を検討する。

#### b. `start-with-env.sh` の改善とPrismaマイグレーション
- **問題:** `npx prisma db push` がデプロイ時に失敗し、手動でのDB操作が必要になる。
- **原因:** 以前の `start-with-env.sh` スクリプトが、DBの準備状況を待たずにマイグレーションを実行しようとしていたため。
- **解決策:**
    1.  `start-with-env.sh` を更新し、`npx prisma db push --skip-generate` を使用するように変更。これにより、マイグレーションのみが実行され、コード生成はスキップされる。
    2.  スクリプトに適切なエラーハンドリングを追加し、マイグレーションが失敗した場合でもコンテナが異常終了しないようにした。
- **教訓:** Dockerコンテナの起動スクリプトでは、依存サービスの準備状況を考慮し、堅牢なエラーハンドリングと適切なコマンドオプション（例: `--skip-generate`）を使用することが重要である。

#### c. Docker Composeのヘルスチェックと起動順序
- **問題:** アプリケーションコンテナがデータベースコンテナの準備が整う前に起動し、DB接続エラーが発生する。
- **原因:** `docker-compose.yml` の設定が、DBコンテナの「サービスが起動した」ことのみを検知し、「サービスが完全に利用可能になった」ことを検知していなかったため。
- **解決策:** `docker-compose.yml` の `db` サービスに `healthcheck` を追加し、`app` サービスの `depends_on` に `service_healthy` 条件を指定することで、DBが完全に利用可能になってから `app` が起動するように制御した。
- **教訓:** `depends_on` はコンテナの起動順序を保証するが、サービスが「利用可能」になることを保証するものではない。データベースなどの依存サービスには `healthcheck` を設定し、`depends_on: service_healthy` を組み合わせることで、より堅牢な起動シーケンスを構築できる。

#### d. ポート競合の再発と解決策
- **問題:** `docker compose up` 実行時に、ポート3000が既に使用されている旨のエラーが頻繁に発生する。
- **原因:** 以前のコンテナや、手動で起動されたNext.jsサーバープロセスが残存していたため。
- **解決策:**
    1.  `ssh conoha-vps "ss -tulnp | grep 3000"` でポートを占有しているプロセスを特定。
    2.  `ssh conoha-vps "pkill -f next-server"` や `ssh conoha-vps "fuser -k 3000/tcp"` を使用して、残存プロセスを強制終了する。
    3.  デプロイ前に `docker compose down -v` を実行し、古いコンテナとボリュームを完全に削除する運用を徹底する。
- **教訓:** ポート競合は頻繁に発生する問題であり、`ss`, `pkill`, `fuser` などのコマンドを使いこなすことが迅速なトラブルシューティングに繋がる。

#### e. 完全自動化されたデプロイプロセスの検証
- **検証内容:** `docker compose down -v` から `docker compose up --build -d` を実行し、以下の項目が自動で完了することを確認した。
    - Dockerイメージのビルド
    - データベースの初期化とPrismaマイグレーションの適用
    - アプリケーションの起動
    - ヘルスチェック（`/api/health` へのアクセス）
    - データベーステーブルの自動生成（`accounts`, `articles`, `comments`, `likes`, `sessions`, `users`, `verification_tokens` の7テーブル）
- **結果:** 全ての項目が自動で正常に完了することを確認。手動でのDB操作は不要となった。
- **教訓:** デプロイプロセスの自動化は、再現性と信頼性を高める上で極めて重要である。特に、クリーンな状態からの完全な再構築（`down -v` からの `up --build`）が自動で成功することは、本番運用における安定性の指標となる。

## 2025年8月18日: 本番環境のOAuth設定 (TSK-045) からの知見

### 1. 概要
本番環境でのOAuth認証設定において、環境変数の読み込み、ポート競合、データベース接続など複数の問題が発生した。これらの問題解決の過程と、得られた教訓を記録する。

### 2. 主要な問題と解決策

#### a. Docker Composeにおける環境変数の読み込み問題
- **問題:** `.env.production` ファイルにOAuth認証情報を設定しても、Dockerコンテナ内で環境変数が正しく読み込まれない、または古い値が参照される。
- **原因:**
    1.  `docker-compose.yml` 内で `env_file` を指定しているにも関わらず、`environment` セクションで同じ環境変数をハードコードしていたため、`environment` の設定が優先されていた。
    2.  `docker compose restart` では、既存のコンテナが再利用されるため、`.env.production` の変更が反映されない場合がある。
- **解決策:**
    1.  `docker-compose.yml` の `environment` セクションから、`.env.production` で管理すべき環境変数のハードコードを削除し、`env_file: .env.production` のみで管理するように統一した。
    2.  環境変数の変更を確実に反映させるため、`docker compose down` で既存のコンテナとネットワークを削除し、`docker compose up -d` で再構築する運用を徹底する。
- **教訓:** Docker Composeで環境変数を管理する際は、`env_file` と `environment` の役割を明確にし、重複や優先順位による意図しない挙動に注意する。変更を確実に反映させるには `down/up` が最も確実な方法である。

#### b. ポート競合の解決
- **問題:** アプリケーション起動時にポート3000が既に使用されており、コンテナが起動できない。
- **原因:** 以前のアプリケーションプロセスや、Dockerコンテナの残骸がポートを占有していたため。
- **解決策:**
    1.  `ssh conoha-vps "ss -tlnp | grep :3000"` でポートを占有しているプロセスを特定。
    2.  特定したプロセスのPID（プロセスID）を確認し、`ssh conoha-vps "kill <PID>"` または `ssh conoha-vps "fuser -k 3000/tcp"` を使用して、残存プロセスを強制終了する。
    3.  `ssh conoha-vps "pkill -f next-server"` のように、プロセス名で関連プロセスを一括終了させることも有効。
- **教訓:** ポート競合は頻繁に発生する問題であり、`ss`, `pkill`, `fuser` などのコマンドを使いこなすことが迅速なトラブルシューティングに繋がる。

#### c. データベース接続問題とデータリセット
- **問題:** アプリケーションがデータベースに接続できない、または認証エラーが発生する。
- **原因:**
    1.  アプリケーションコンテナがDBコンテナの準備が整う前に接続を試みていた。
    2.  過去のDBデータや設定が残っており、新しい認証情報と競合していた。
- **解決策:**
    1.  `docker compose down -v` を実行し、コンテナだけでなく関連するDockerボリューム（DBデータが永続化されている場所）も完全に削除してから再構築した。これにより、DBがクリーンな状態で再初期化され、認証情報が正しく適用された。
    2.  アプリケーションコンテナがDBコンテナの起動を待つように、`docker-compose.yml` に `depends_on` や `healthcheck` を適切に設定することが恒久的な解決策となる（TSK-046で対応予定）。
- **教訓:** データベース関連の問題で認証エラーや接続エラーが続く場合、特に開発・テスト環境では、`docker compose down -v` によるボリュームの完全リセットが迅速な解決策となる場合がある。ただし、本番環境での実行はデータ損失のリスクがあるため、慎重に行う必要がある。

#### d. コンテナ内の環境変数確認
- **問題:** 環境変数がコンテナ内に正しく渡されているか確認したい。
- **解決策:** `ssh conoha-vps "docker exec <コンテナ名またはID> env | grep <環境変数名の一部>"` を使用して、実行中のコンテナ内部の環境変数を確認した。
    - 例: `ssh conoha-vps "docker exec uch-app env | grep GOOGLE_CLIENT_"`
- **教訓:** 環境変数の問題はデバッグが困難な場合があるため、実際にコンテナ内部でどのような環境変数が設定されているかを確認するこのコマンドは非常に有用である。

## 2025年8月18日: VPS初回デプロイ(TSK-043)からの知見

### 1. 概要
VPS(ConoHa)への初回デプロイ作業において、ビルドエラー、データベース接続、ポート競合など、複数の問題が発生した。これらの問題を解決した際のトラブルシューティング手順を記録する。

### 2. トラブルシューティング・サマリー

#### a. Next.jsビルド関連の問題

- **Google Fontsのタイムアウト**
    - **症状:** `next build` 中にGoogle Fontsのダウンロードでタイムアウトし、ビルドが失敗する。
    - **対策1:** `next.config.js` に `optimizeFonts: false` を追加し、フォント最適化を無効化する。
    - **対策2:** 対策1でも解決しない場合、問題となっている特定のWebフォント(`Shippori_Mincho_B1`など)の読み込みを `src/app/layout.tsx` から削除し、代替のフォントで対応する。

#### b. Dockerおよびデータベース関連の問題

- **`docker compose` コマンド**
    - **症状:** `docker-compose` (ハイフンあり) コマンドが見つからないエラー。
    - **対策:** VPSにインストールされているDockerはCompose V2であるため、`docker compose` (スペース区切り) を使用する。

- **ポート競合によるコンテナ起動失敗**
    - **症状:** `docker compose up` 実行時に、ポート3000が既に使用されている旨のエラー(`address already in use`)が表示される。
    - **対策:** `ssh conoha-vps "ss -tlnp | grep :3000"` などでポートを使用しているプロセスID(PID)を特定し、`kill -9 <PID>` でプロセスを強制終了させる。`pkill -f next-server` のようにプロセス名で強制終了することも有効。

- **DBマイグレーションの失敗**
    - **症状:** `app`コンテナの起動スクリプト(`start-with-env.sh`)内で実行される `npx prisma db push` がDB接続エラーで失敗する。
    - **原因:** `app`コンテナがDBコンテナの準備が整う前に起動してしまうため。
    - **暫定対応:** `docker compose exec db createdb -U <user> <dbname>` でDBを手動作成し、`psql`で`CREATE TABLE`文を直接実行した。
    - **恒久対策(推奨):** `docker-compose.yml`の`app`サービスに`depends_on`と`healthcheck`を組み合わせた起動順序制御を追加する。(→**TSK-046**で対応)

#### c. VPS環境設定

- **デプロイ直後の外部アクセス失敗**
    *   **症状:** コンテナは正常に起動しているが、外部からブラウザや`curl`でアクセスできない(タイムアウトする)。
    *   **原因特定:** 当初、VPSのファイアウォール(UFW)がポート3000をブロックしていると想定された(TSK-044で要確認とされた)。しかし、調査の結果、ファイアウォール設定は既に正しくポート3000を許可していた。根本的な原因は、Next.jsアプリケーションの起動に時間がかかり、疎通確認のタイミングではまだ完全にリクエストを受け付けられる状態になっていなかったためと判断される。
    *   **対策/教訓:** デプロイ直後に外部アクセスが失敗した場合、まずファイアウォール設定(`sudo ufw status`)を確認することは定石だが、それで問題がない場合は、数分待ってから再度アクセスを試みる、または `docker compose logs -f app` でアプリケーションの起動ログを監視し、完全に起動したことを確認してから疎通確認を行うべきである。
