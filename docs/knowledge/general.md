## 2025年8月17日: 新規記事投稿APIの不具合対応 (TSK-036) からの知見

### 1. インシデント概要
- **問題:** 新規記事投稿ページで、ヒーロー画像が保存されず、下書き機能が動作しない不具合が発生。
- **根本原因:** フロントエンドとバックエンド間でのAPI契約の不一致。具体的には、リクエストペイロードのJSONキー名が異なっていた (`heroImage` vs `heroImageUrl`)。

### 2. 教訓と再発防止策

#### a. API契約の文書化と合意形成の徹底
- **教訓:** 口頭や非公式な合意のみで開発を進めると、フロントエンドとバックエンドの実装に乖離が生まれるリスクが非常に高い。
- **対策:** APIを新規設計・変更する際は、必ず**API仕様書**（エンドポイント、HTTPメソッド、リクエスト/レスポンスのペイロード、ステータスコードなど）をドキュメントとして作成する。そして、関係者（フロントエンド担当、バックエンド担当）がその仕様書に合意してから実装に着手するプロセスを徹底する。

#### b. 型定義の共有による契約のコードレベルでの強制
- **教訓:** 人間の注意力には限界があり、タイポや記憶違いによるキー名の不一致は容易に発生する。
- **対策:** 可能であれば、APIのデータ構造に関する型定義（TypeScriptの`interface`や`type`）を、フロントエンドとバックエンドで共有する。
    - 例えば、`src/types/api/article.ts` のようなファイルを作成し、`ArticleCreationRequest` のような型を定義する。
    - これにより、コンパイル時に型の不一致を検知でき、実行時エラーを未然に防ぐことができる。

#### c. 開発サイクルの最終段階における統合テストの必須化
- **教訓:** 各モジュール（フロントエンド、バックエンド）の単体テストだけでは、モジュール間の連携ミスは見逃される。
- **対策:** 機能開発の完了を判断する前に、必ずフロントエンドとバックエンドを結合した状態での**統合テスト**（E2Eテスト、または手動でのシナリオテスト）を実施する。これにより、「フォームに入力してボタンを押し、データが正しくDBに保存される」という一連の流れ全体が正しく機能することを確認する。

## 2025年8月17日: Missing dependencies (不足している依存関係)

- **問題**: `react-markdown`, `rehype-raw`, `rehype-sanitize`, `remark-gfm` といったコンポーネントやライブラリが不足しており、ビルドエラーの原因となっていた。
- **対応策**: 以下のコマンドで不足している依存関係をインストールした。
    `npm install react-markdown rehype-raw rehype-sanitize remark-gfm`
- **関連ファイル**: `package.json`
- **知見**: プロジェクトの移行やクローン時には、`package.json` に記載されている依存関係がすべてインストールされているか、`npm install` または `yarn install` を実行して確認することが不可欠である。特に、特定の機能が動作しない場合は、関連する依存関係の不足を疑うべきである。

## 2025年8月19日: 記事更新失敗トラブルシューティング (TSK-067) からの知見

### 1. インシデント概要
- **問題:** 本番環境 (https://uchinokiroku.com/) での記事更新が完了しない不具合。更新ボタンクリック後、画面遷移が発生せず、記事が保存されない。
- **影響範囲:** 記事編集機能全般

### 2. 技術分析結果

#### API実装の確認 (`src/app/api/articles/[slug]/route.ts`)
- **PUTメソッド (51-139行目):** 記事更新処理の実装を確認
- **認証チェック:** NextAuth セッション + 開発環境向けダミーユーザー対応
- **権限チェック:** ゲストユーザーの書き込み制限 + 記事所有者チェック
- **バリデーション:** タイトル・本文の必須チェック
- **エラーハンドリング:** try-catch + console.error によるログ出力

#### フロントエンド実装の確認 (`src/app/articles/[slug]/edit/page.tsx`)
- **handleSubmit関数 (116-162行目):** PUT リクエストの送信処理
- **認証チェック:** ユーザーとauthorの email 比較による権限確認 (82行目)
- **エラーハンドリング:** response.ok チェック + エラーメッセージ表示
- **リダイレクト:** 成功時に記事詳細ページへ遷移 (155行目)

### 3. 推定される問題パターンと対処法

#### パターン1: 認証セッション問題（最も可能性が高い）
- **原因:** NextAuth セッションの期限切れまたは不整合
- **確認方法:** 
  - ブラウザのデベロッパーツール → アプリケーションタブ → Cookies で next-auth 関連の情報確認
  - コンソールで認証エラーメッセージの確認
- **対処法:** ログアウト → 再ログイン

#### パターン2: サーバーサイドエラー
- **原因:** データベース接続問題、Prisma更新処理のエラー
- **確認方法:** デベロッパーツール → ネットワークタブで API レスポンス（500 Internal Server Error）を確認
- **対処法:** サーバーログの確認と技術的対応が必要

#### パターン3: CORS またはネットワーク問題
- **原因:** クロスオリジン制約、ネットワーク接続不安定
- **確認方法:** コンソールタブで CORS エラーメッセージを確認
- **対処法:** ブラウザ再読み込み、別ブラウザでの試行

### 4. 推奨トラブルシューティング手順

#### Phase 1: 即座に実行可能（エンドユーザー対応）
1. **ブラウザ操作による基本対処**
   - ページ再読み込み（Ctrl+F5 または Cmd+R）
   - ログアウト → 再ログイン
   - 別ブラウザ（Chrome、Edge、Firefox）での試行
   - シークレット/プライベートモードでの試行

2. **デベロッパーツールを使った原因調査**
   ```
   手順:
   1. F12 でデベロッパーツール開く
   2. ネットワークタブに移動
   3. 記事編集画面で「変更を保存」ボタンクリック
   4. PUT /api/articles/[slug] リクエストの確認
      - ステータスコード（200=成功、401=認証、403=権限、500=サーバーエラー）
      - レスポンス内容の確認
   5. コンソールタブでエラーメッセージ確認
   ```

#### Phase 2: 技術的対応（開発者向け）
1. **サーバーログ分析**

### 5. 予防策と改善提案

#### ユーザビリティ向上
- **温かいエラーメッセージ:** 「うちのきろく」の設計思想に基づき、技術的なエラーよりも「あれれ？うまく保存できませんでした。少し待ってからもう一度お試しください😊」のような表現を検討

#### 技術的改善
- **リトライ機能:** 一時的なネットワークエラーに対する自動再試行

- **ログ改善:** より詳細なエラー情報の記録（ユーザーID、記事ID、エラー詳細）
