# 家族ブログ「uchinokiroku.com」メディア管理機能 設計書

## 1. 目的

家族が気軽に写真や動画を共有し、長期的に思い出を保存・閲覧できる仕組みを構築する。アップロードの手間を最小限にし、管理はAI執事が自動で行う。

## 2. アーキテクチャ概要

- **フロントエンド:** Next.js
- **バックエンド:** Next.js API Routes
- **データベース:** PostgreSQL (ConoHa VPS上)
- **ファイルストレージ:** Cloudflare R2
- **バックグラウンド処理:** AI執事 (Node.jsスクリプトをVPS上で定期実行)

### データフロー
1.  **アップロード:** ユーザーがブラウザからファイルを選択。Next.jsサーバーがR2の署名付きURLを発行し、ブラウザは直接R2にファイルをアップロードする。
2.  **DB記録:** アップロード情報（ファイル名、アップロード主など）をPostgreSQLに保存。
3.  **自動処理:** AI執事がDBを監視。未処理のファイルを見つけ、R2からダウンロード → 最適化 → R2に再アップロード → DB情報を更新する。
4.  **表示:** ユーザーがギャラリーを開くと、Next.jsはDBからファイル情報を取得し、R2上の最適化済みファイルを表示する。



## 3. データベース設計 (PostgreSQL)

### `media` テーブル
アップロードされたメディアファイル（画像・動画）の基本情報を管理する。

| カラム名 | データ型 | 説明 | 例 |
| :--- | :--- | :--- | :--- |
| `id` | `UUID` | 主キー。ランダムなID | `a1b2c3d4-...` |
| `uploader_id` | `VARCHAR(255)` | アップロードした人のID | `user_01` |
| `original_filename`| `TEXT` | 元のファイル名 | `IMG_1234.JPG` |
| `storage_key` | `TEXT` | R2に保存する際のユニークなキー | `a1b2c3d4-...` |
| `mime_type` | `VARCHAR(100)`| ファイルのMIMEタイプ | `image/jpeg` |
| `status` | `VARCHAR(50)` | 処理ステータス | `pending`, `optimized`, `error` |
| `created_at` | `TIMESTAMPTZ` | 作成日時 | `2025-08-21 ...` |

### `media_optimized` テーブル
最適化されたファイルの情報を管理する。

| カラム名 | データ型 | 説明 |
| :--- | :--- | :--- |
| `id` | `UUID` | 主キー |
| `media_id` | `UUID` | `media`テーブルへの外部キー |
| `storage_key` | `TEXT` | 最適化後のファイルのキー |
| `mime_type` | `VARCHAR(100)`| `image/webp` など |
| `width` | `INT` | 画像の幅 |
| `height` | `INT` | 画像の高さ |
| `created_at` | `TIMESTAMPTZ` | 作成日時 |

## 4. APIエンドポイント設計

- **`POST /api/media/generate-upload-url`**
  - **役割:** R2へのアップロード用署名付きURLを発行する。
  - **リクエスト:** `{ "fileName": "IMG_1234.JPG", "fileType": "image/jpeg" }`
  - **レスポンス:** `{ "url": "https://...", "storageKey": "a1b2c3d4-..." }`

- **`POST /api/media/upload-complete`**
  - **役割:** アップロード完了後、DBに初期レコードを作成する。
  - **リクエスト:** `{ "storageKey": "a1b2c3d4-...", "originalFilename": "IMG_1234.JPG", ... }`
  - **レスポンス:** `{ "success": true }`

- **`GET /api/media`**
  - **役割:** ギャラリーに表示するメディア一覧を取得する。
  - **レスポンス:** `[ { "id": "...", "optimizedUrl": "...", ... } ]`

## 5. AI執事の処理ロジック

- 5分ごとにcronで起動。
- `media`テーブルで`status`が`pending`のレコードを10件取得。
- 1件ずつ以下の処理を実行:
    1.  R2から`originals/` + `storage_key`のファイルをダウンロード。
    2.  `sharp`（画像）や`ffmpeg`（動画）で最適化処理。
    3.  `optimized/` + `storage_key`としてR2にアップロード。
    4.  `media_optimized`テーブルにレコードを作成。
    5.  `media`テーブルの`status`を`optimized`に更新。
- エラーが発生したら`status`を`error`にしてログを記録。
