## 1. 本書の位置づけと目的

本書は、「うちのきろく」プロジェクトにおけるソフトウェア品質を保証するためのテスト戦略を定義するものです。

**目的:**
- プロダクトの信頼性を高め、ユーザーに安心して利用してもらうこと。
- 機能追加やリファクタリングを安全かつ迅速に行えるようにすること。
- バグの早期発見と手戻りコストの削減を実現すること。

## 2. テスト戦略の全体像

本プロジェクトでは、性質の異なるテストを組み合わせた「テストピラミッド」の考え方を採用し、アプリケーションの品質を多角的に保証する。

- **静的解析 (Static Analysis):** コーディング段階での品質確保
- **単体テスト (Unit Tests):** 個々の部品の品質確保
- **結合テスト (Integration Tests):** 部品間の連携の品質確保
- **E2Eテスト (End-to-End Tests):** ユーザーシナリオ全体の品質確保

## 3. 各テストの詳細

### 3.1. 静的解析 (Static Analysis)

- **目的:** コードを実行する前に、潜在的なバグ、コード規約違反、型エラーを自動的に検出する。
- **ツール:**
    - **TypeScript Compiler (`tsc`):** 型安全性の保証。
    - **ESLint:** コードの品質と一貫性の維持。
    - **Prettier:** コードフォーマットの自動整形。
- **運用ルール:**
    - コミット前に必ず全ての静的解析を実行し、エラーがない状態を維持する。
    - (将来的には) CI/CDパイプラインに組み込み、自動化する。

### 3.2. 単体テスト (Unit Tests)

- **目的:** 個別の関数、コンポーネント、APIルートハンドラなど、最小単位のロジックが仕様通りに正しく動作することを確認する。
- **対象:**
    - `lib` や `utils` 内のヘルパー関数。
    - 複雑なロジックを持つReactコンポーネントの内部関数。
    - APIルートハンドラのビジネスロジック部分。
- **ツール候補:**
    - **Jest:** 業界標準でエコシステムが豊富。
    - **Vitest:** Viteベースで高速に動作。
- **基本方針:**
    - 外部依存（API、DB）はモック化する。
    - 純粋な入力と出力に焦点を当て、副作用をテストしない。
    - カバレッジ目標: **70%** を目指す。

### 3.3. 結合テスト (Integration Tests)

- **目的:** 複数のコンポーネントや、フロントエンドとバックエンドAPIの連携部分が、意図通りに動作することを確認する。
- **対象:**
    - フォームコンポーネントと、それを呼び出すページの連携。
    - データ取得コンポーネントと、実際にAPIを呼び出す処理の連携。
    - 状態管理（Context）と、それを利用するコンポーネント群の連携。
- **ツール候補:**
    - **React Testing Library:** ユーザーの視点に近いテストが可能。
    - **Mock Service Worker (MSW):** APIリクエストをインターセプトし、モックサーバーとして振る舞う。
- **基本方針:**
    - ユーザーのインタラクション（クリック、入力など）を起点にテストを記述する。
    - APIのモック化にはMSWを利用し、より実践的なテストを行う。

### 3.4. E2Eテスト (End-to-End Tests)

- **目的:** ユーザーの一連の操作シナリオ全体が、実際のブラウザ上で問題なく動作することを確認する。
- **対象:**
    - **最重要シナリオ:**
        1.  ログイン → 記事作成 → 記事詳細表示 → ログアウト
        2.  記事一覧表示 → 記事編集 → 更新内容の確認
        3.  記事一覧表示 → 記事削除 → 一覧から消えることの確認
- **ツール候補:**
    - **Playwright:** Microsoft製。高速で多機能。
    - **Cypress:** 長い実績と豊富なドキュメント。
- **基本方針:**
    - 主要なユーザーストーリー（クリティカルパス）に絞ってテストケースを作成する。
    - テストは、実際のユーザー操作を忠実に再現するように記述する。
    - (将来的には) CI/CDパイプラインに組み込み、デプロイ前に必ず実行する。

## 4. 今後のステップ

1.  **ツール選定の確定:** 各テスト層で使用するツールを正式に決定する。
2.  **環境構築:** 選定したテストツールをプロジェクトにインストールし、設定を行う。
3.  **テスト記述:** まずは主要な機能（記事CRUD）から、各テスト層のテストを記述していく。
4.  **CI連携:** 将来的に、GitHub ActionsなどのCIツールと連携し、テストの自動実行を目指す。
